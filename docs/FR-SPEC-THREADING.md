# FR: Spec Threading — Init → Terminal Data Flow

## Problem

Init and terminal intent gates are currently independent:
- **Init gate** validates plan clarity (abstract intent checks)
- **Terminal gate** validates output correctness (visual observations)

But there's no connection between them. Terminal gate doesn't know "what was I supposed to build?" It validates that the app launches and features exist, but has no reference to what the plan promised.

This creates a gap: if init gate clarified a vague plan into concrete requirements (e.g., "database layer must have CRUD UI"), the terminal gate should verify those specific requirements, not just generic app health.

## Solution

**Spec-clarified.json**: A machine-readable contract generated by init gate and verified by terminal gate.

```json
{
  "features": [
    {
      "id": "crud-add",
      "selector": "input[placeholder*=Add]",
      "observation": "visible",
      "evidence": "Plan required concrete add button, selector derived from WCAG accessibility path"
    },
    {
      "id": "crud-toggle",
      "selector": "input[type=checkbox]",
      "observation": "interactive",
      "evidence": "Todo toggle must be keyboard-accessible per plan spec"
    },
    {
      "id": "text-contrast",
      "minRatio": 4.5,
      "observation": "contrast",
      "evidence": "Accessibility requirement from plan clarity gate"
    }
  ],
  "gaps": [],
  "confidence": 0.98,
  "generated": "2026-02-27T05:58:00Z"
}
```

## Architecture

### Data Flow
```
vague plan
  ↓
[INIT GATE: validatePlanClarity]
  ↓
extracts clarity gaps:
  - VagueProduces: "database" → ["auth.ts", "db.ts", "api.ts"]
  - UnresolvableConsumes: resolved
  - NoValidate: rules added
  - OwnershipConflict: resolved
  - BroadScope: decomposed
  ↓
[generates spec-clarified.json]
  ↓
spec-clarified.json (produced artifact)
  ↓
[TERMINAL GATE: validateAgainstContract]
  ↓
[EXPLORE SCRIPT: observe features & validate]
  ↓
observations matched against contract:
  - Is input[placeholder*=Add] visible? ✅
  - Is input[type=checkbox] interactive? ✅
  - Is text contrast ≥ 4.5:1? ✅
  ↓
contract satisfied → expansion unnecessary
```

### DAG Integration

**Init Gate Node** (produces):
```typescript
{
  id: 'plan-clarity',
  desc: '...',
  produces: ['spec-clarified.json'],  // ← NEW
  validate: [{
    type: 'intent',
    statement: 'Plan is unambiguous and ready to execute',
    expandOnFail: true,
  }],
}
```

**Terminal Gate Node** (consumes):
```typescript
{
  id: 'term',
  desc: '...',
  produces: [],
  consumes: ['spec-clarified.json'],  // ← NEW, makes dependency explicit
  deps: [...],
  validate: [{
    type: 'runtime-explore',
    script: 'scripts/explore-validate-contract.ts',
    specPath: 'spec-clarified.json',  // ← context for script
  }],
}
```

When `roadmap propagate` runs, it auto-derives that terminal gate requires `spec-clarified.json` as input. No manual bookkeeping.

## Implementation

### 1. Spec Generation (init-gate side)

**Function**: `generateClarifiedSpec(dag, clarityGaps)`

Inputs:
- DAG structure
- PlanClarityGap[] from validatePlanClarity

Algorithm:
```
For each feature in the plan (extracted from node produces/validate):
  For each gap that affects it:
    - VagueProduces: generate selector heuristics from file paths
    - UnresolvableConsumes: skip (resolved by propagate)
    - BroadScope: decompose into individual features

Generate observations to check:
  - visible: for UI elements (inputs, buttons)
  - interactive: for interactive elements (forms)
  - toggles-class: for state changes
  - contrast: for accessibility (WCAG AA 4.5:1)
  - count: for lists (expected ≥ 1)
```

Output: SpecClarifiedJson

### 2. Spec Verification (terminal-gate side)

**Function**: `verifyObservationsAgainstContract(spec, observations)`

Inputs:
- SpecClarifiedJson (from init gate)
- ObservationResult[] (from explore script)

For each feature in spec:
1. Match observation by ID
2. Check pass/fail
3. Compare evidence (e.g., actual contrast ratio vs minRatio)

Output: VerificationResult
```typescript
{
  passed: boolean,
  matched: number,
  failed: number,
  failures: Array<{ id, expected, actual, evidence }>,
}
```

### 3. Explore Script Integration

**Script**: `explore-validate-contract.ts`

Reads:
- `process.env.SPEC_PATH` → loads spec-clarified.json
- `process.env.CDP_URL` → connects to app

For each feature in spec:
1. Call appropriate observation helper (checkVisible, checkInteractive, etc.)
2. Emit ObservationResult with evidence
3. Verify against contract expectation

```typescript
import { connectAndFindPage, checkVisible, checkContrast } from 'roadmap/explore'
import { readFileSync } from 'fs'

const spec = JSON.parse(readFileSync(process.env.SPEC_PATH!))
const { page, browser } = await connectAndFindPage(process.env.CDP_URL)

const obs = await Promise.all(
  spec.features.map(feature => {
    switch (feature.observation) {
      case 'visible': return checkVisible(page, feature.selector, feature.id)
      case 'contrast': return checkContrast(page, feature.selector, 'body', feature.minRatio, feature.id)
      // ...
    }
  })
)

console.log(JSON.stringify({ observations: obs }))
await browser.close()
```

## Tests

### Spec Generation Tests
- Gap → feature selector conversion
- VagueProduces generates concrete paths
- BroadScope decomposed into individual features
- Confidence calculation

### Spec Verification Tests
- Passing observations match contract
- Failing observations detected
- Evidence comparison (ratios, values)
- Unmatched observations flagged

### E2E Tests
- Vague plan → clarified spec → verified output
- Mock app with features matching spec
- Explore script validates against contract
- Terminal gate passes/fails based on contract match

## Validation

Acceptance scenarios from FR-SPEC-THREADING.md:

### Spec Generation
**Given** a vague plan with unclear produces ("database")
**When** init gate validates clarity
**Then** spec-clarified.json is generated with concrete features (auth.ts, db.ts, api.ts)

### Contract Verification
**Given** a spec contract with feature selectors
**When** terminal gate runs explore script
**Then** observations are validated against contract selectors
**And** missing/failing observations trigger expansion

### Terminal Integration
**Given** terminal gate consumes spec-clarified.json
**When** roadmap propagate runs
**Then** spec-clarified.json is auto-required as input
**And** no manual depends bookkeeping needed

## Scope

**In scope:**
- `generateClarifiedSpec()` function
- `verifyObservationsAgainstContract()` function
- Explore script template with contract validation
- Tests (spec-contract, spec-verifier, e2e)
- Documentation (this file)

**Out of scope:**
- Visual diffing / screenshot comparison
- Selector auto-generation from actual DOM (spec features define selectors manually)
- Multi-page app handling (single page assumption)
