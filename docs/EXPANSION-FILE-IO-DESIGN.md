# Expansion Script File I/O Design

## Overview

Plan mode nodes (declared with `mode: 'plan'`) decompose into child nodes via DAG expansion. This design covers the file I/O system for:

1. **Writing** expansion scripts to disk (generated by intent-expansion, written before `roadmap expand` executes)
2. **Reading** expansion scripts (CLI invocation, validation, commit)
3. **Auditability** and history tracing (user review, forensics)

The goal is to make expansions human-reviewable, deterministic, and fully traceable from plan node failure → diagnosis → script generation → committed DAG.

---

## 1. File Structure

### Directory Layout

```
.roadmap/
├── head.json                           # Active DAG
├── expansions/                         # All expansion scripts (history)
│   ├── <nodeId>-<timestamp>.ts        # One script per invocation
│   ├── approve-phase-2-1709078400.ts  # approval plan 2025-02-27 04:00:00
│   ├── approve-phase-2-1709078401.ts  # rerun (same parent, slightly different)
│   └── runtime-explore-1709078402.ts  # runtime exploration output
├── memory/
├── checkpoints/
└── ...
```

### Naming Convention

**Pattern:** `<nodeId>-<timestamp>.ts`

- `nodeId`: kebab-case node ID of the plan node that triggered expansion
- `timestamp`: Unix seconds when script was written (ISO 8601 sortable, UTC)
- `.ts`: Always TypeScript, executable with `node --experimental-strip-types`

**Examples:**
- `plan-auth-strategy-1709078400.ts` (Feb 27 2025, 04:00:00 UTC)
- `design-api-schema-1709078401.ts` (1 second later)
- `runtime-explore-1709078402.ts` (runtime-explore diagnostics)

**Rationale:**
- Timestamp avoids collisions across parallel agents or retries
- Node ID provides semantic context for grep/audit
- Extensions clear (TypeScript, not JSON)
- Alphabetical sort = chronological order within a node's history

### Cleanup Policy

**Keep-all (default):** Store every expansion script ever written. Scripts are small (typically <10 KB), and history is forensic value.

- `.roadmap/expansions/` grows with N plan nodes × M expansion rounds
- No auto-cleanup; user can `git rm .roadmap/expansions/*-old.ts` if needed
- Checkpoints (`.roadmap/checkpoints/`) already use keep-all; expansions follow same pattern

**Garbage collection trigger (optional, future):**
- If expansions/ > 100 MB: warn user, suggest archiving to `.roadmap/expansions.tar.gz`
- Can add `roadmap cleanup --expansions` later if needed

## 2. Content Format

### File Structure

Every expansion script is valid, executable TypeScript. Structure:

```typescript
#!/usr/bin/env node
// Expansion script for: <nodeId>
// Generated: <ISO 8601 timestamp>
// Parent: <parentNodeId>
// Diagnosis: <one-line summary of why expansion occurred>

import { readFileSync, writeFileSync } from 'node:fs';
import { join } from 'node:path';
import type { FixNodeSpec } from '../src/lib/intent-expansion.ts';

const headPath = join(process.cwd(), '.roadmap', 'head.json');
const dag = JSON.parse(readFileSync(headPath, 'utf-8'));

// ─────────────────────────────────────────────────────────
// Expansion: <section name>
// ─────────────────────────────────────────────────────────

// Append nodes to dag.nodes
dag.nodes['<nodeId>-fix-0'] = {
  id: '<nodeId>-fix-0',
  desc: 'Fix: <intent statement> (confidence 0.42/0.90)',
  expandedFrom: '<parentNodeId>',
  produces: ['src/foo.ts'],
  consumes: ['src/bar.ts'],
  ambient: ['docs/spec.md'],
  deps: ['<parentNodeId>'],
  validate: [
    {
      type: 'intent',
      statement: 'auth module implements JWT rotation',
      confidence: 0.90,
      evaluator: 'self',
      expandOnFail: true,
      maxExpansionDepth: 3,
    },
    {
      type: 'artifact-exists',
      target: 'src/foo.ts',
    },
  ],
  idempotent: true,
  _intentDiagnosis: {
    statement: 'auth module implements JWT rotation',
    achievedConfidence: 0.42,
    threshold: 0.90,
    reasoning: 'Token rotation logic present but not tested; confidence depends on test coverage.',
    evidence: ['src/auth.ts:45', 'tests/auth.test.ts (no rotation tests)'],
    expansionDepth: 1,
  },
};

// ─────────────────────────────────────────────────────────
// Connect: rewire parent deps
// ─────────────────────────────────────────────────────────

// If parent is a plan node, remove it from deps and add fix nodes
// (Plan nodes don't execute; their expanded children do)
dag.nodes['<parentNodeId>'] = {
  ...dag.nodes['<parentNodeId>'],
  deps: ['<nodeId>-fix-0'],  // formerly []
};

// Write back
writeFileSync(headPath, JSON.stringify(dag, null, 2) + '\n');
console.log('Expanded: <nodeId> → <nodeId>-fix-0 (+1 node)');
```

### Provenance Encoding

**Attached to each fix node:**

```typescript
_intentDiagnosis: {
  statement: 'Feature: JWT refresh token rotation',
  achievedConfidence: 0.42,        // actual measured confidence
  threshold: 0.90,                 // required confidence
  reasoning: 'Tokens rotate but no...', // from IntentJudgment
  evidence: ['src/auth.ts:45'],    // what evidence informed this
  expansionDepth: 1,               // first expansion (0 = original intent)
}
```

**Attached to each child node:**

```typescript
expandedFrom: '<parentNodeId>'  // provenance link back to plan node
```

**In consumes:**

```typescript
consumes: ['src/parent-output.ts']  // reads parent's produces
```

This creates a full audit trail:
- Plan node X failed intent Y → expansion created
- Fix nodes Z1, Z2, Z3 created with `expandedFrom: 'X'`
- Each fix node's `_intentDiagnosis` explains why it exists
- Each fix node's `consumes` shows data dependency on parent

## 3. Integration Points

### A. Write Trigger: `generateIntentExpansion()` → `writeExpansionScript()`

**Flow:**

```
cmdComplete() runs validation
  → intent fails, expandOnFail=true
  → extractIntentFailures() → failures[]
  → generateIntentExpansion() → FixNodeSpec[]
  → writeExpansionScript(parentId, failures, fixNodes)  [NEW]
    → .roadmap/expansions/<nodeId>-<timestamp>.ts
  → output path in JSON response
  → user reviews then runs: roadmap expand <path>
```

**New function signature:**

```typescript
// src/lib/expansion-writer.ts
export function writeExpansionScript(
  parentId: string,
  parentNode: NodeSpec<any>,
  failures: IntentFailure[],
  fixNodes: FixNodeSpec[],
  reason: 'intent-expansion' | 'runtime-explore' | 'escalation-recovery',
  repoRoot: string,
): string {
  // Returns: /abs/path/to/.roadmap/expansions/<nodeId>-<timestamp>.ts
  // Writes: complete, executable TypeScript
}
```

**Placement:**
- New file: `src/lib/expansion-writer.ts` (15–20 lines header, ~80 lines body)
- Exports: `writeExpansionScript`
- Used by: `cmdComplete()` in `bin/roadmap.ts` before JSON response
- Also used by: intent-expansion convergence loop (future)

### B. CLI Invocation: `roadmap expand <path>`

**Current behavior (no change needed):**

```bash
roadmap expand .roadmap/expansions/approve-phase-2-1709078400.ts
```

1. Resolves path (already does this)
2. Validates script exists
3. Snapshots DAG before (node count)
4. Executes: `node --experimental-strip-types <path>`
5. Snapshots DAG after
6. Validates (define, check, verify, terminal intent gate)
7. Commits: `roadmap: expand — +N nodes via <script-path>`
8. Records trail entry with `detail: { script, added, commit }`

**Enhancement (optional, nice-to-have):**

Add `--dry-run` flag:

```bash
roadmap expand <path> --dry-run
```

- Executes script in temp copy of DAG
- Prints what would be added + validation summary
- Does NOT commit
- Useful for user review before `roadmap expand` (final commit)

### C. Output to User (JSON response from `cmdComplete`)

**Before expansion:**

```json
{
  "completed": true,
  "node": "plan-auth-strategy",
  "time_ms": 234
}
```

**After expansion (intent failure with `expandOnFail=true`):**

```json
{
  "completed": false,
  "node": "plan-auth-strategy",
  "validation": {
    "passed": false,
    "expandable": true,
    "script": ".roadmap/expansions/plan-auth-strategy-1709078400.ts",
    "failedIntents": [
      {
        "statement": "JWT module implements rotation",
        "achieved": 0.42,
        "threshold": 0.90,
        "reasoning": "Tokens rotate but no test coverage"
      }
    ],
    "nextStep": "Review the expansion script, then run: roadmap expand .roadmap/expansions/plan-auth-strategy-1709078400.ts"
  }
}
```

## 4. Auditability

### Full Forensic Trail

**Query 1: What expansions has node X undergone?**

```bash
ls -ltr .roadmap/expansions/ | grep "^approve-phase-2"
```

Output:

```
-rw-r--r-- 1 griffin griffin 3456 Feb 27 04:00 approve-phase-2-1709078400.ts
-rw-r--r-- 1 griffin griffin 3512 Feb 27 04:01 approve-phase-2-1709078401.ts  (retry, different intents)
```

**Query 2: Why was this expansion created?**

```bash
head -10 .roadmap/expansions/approve-phase-2-1709078400.ts
```

Output:

```
// Expansion script for: approve-phase-2
// Generated: 2025-02-27T04:00:00Z
// Parent: plan-approvals
// Diagnosis: 'Review latency' intent at depth 1 did not converge
```

**Query 3: What's in the expansion?**

```bash
grep "^dag.nodes\[" .roadmap/expansions/approve-phase-2-1709078400.ts
```

**Query 4: Trace one node's creation back to original failure:**

```bash
# Which expansion created this node?
git log --all --grep="approve-phase-2-fix-0" --oneline
# Find commit: "roadmap: expand — +1 nodes via .roadmap/expansions/approve-phase-2-1709078400.ts"

# Find the script in that commit
git show <commit>:.roadmap/expansions/approve-phase-2-1709078400.ts | grep -A 20 "approve-phase-2-fix-0"

# See _intentDiagnosis block
git show <commit>:.roadmap/expansions/approve-phase-2-1709078400.ts | grep -A 10 "_intentDiagnosis"
```

### Auditability Checklist

- [x] Every fix node has `_intentDiagnosis` with statement, achieved/threshold, reasoning, evidence
- [x] Every fix node has `expandedFrom` pointing to parent plan node
- [x] Every fix node's `consumes` includes parent's `produces`
- [x] Every script file has header comment (nodeId, timestamp, parent, diagnosis)
- [x] Scripts are in `.roadmap/expansions/` (dedicated directory)
- [x] Scripts are committed with `roadmap expand` (git history)
- [x] Scripts are human-readable TypeScript (not binary, not minified)
- [x] Trail entries record script path + nodes added

## 5. Edge Cases

### 5.1 Overwrite Policy

**Scenario:** User runs `cmdComplete` twice on same node, same parent intent failure.

**Current behavior:**
- `writeExpansionScript()` uses timestamp → new file `<nodeId>-<ts2>.ts`
- Two scripts exist: `...-1709078400.ts` and `...-1709078401.ts`
- User manually deletes old one or commits both (usually deletes before expand)

**Rationale:** Scripts are idempotent structural specs. Dupes are harmless. History is forensic value.

**Alternative (not chosen):** Overwrite with `--force`. Too destructive; keeps simpler semantics: one file = one invocation.

### 5.2 Concurrent Expansions from Parallel Agents

**Scenario:** Two agents both hit expandable failures on different plan nodes.

**Behavior:**
- Agent A: `writeExpansionScript()` → `.roadmap/expansions/node-a-1709078400.ts`
- Agent B: `writeExpansionScript()` → `.roadmap/expansions/node-b-1709078400.ts` (different node, same ts is fine)
- Both return paths in JSON responses
- Both commit if users run `roadmap expand` in parallel

**Git conflict?** No. Each script modifies only `head.json` once committed. Git sees:

```
commit A: roadmap: expand — +N1 nodes via node-a-1709078400.ts
commit B: roadmap: expand — +N2 nodes via node-b-1709078400.ts
DAG merges (topo merge if there are common deps)
```

If both try to re-root same terminal node, that's a real conflict but rare (different plan nodes usually target different parts of DAG).

**Mitigation (if needed):** `roadmap expand <path> --no-rebase` fails if upstream DAG changed. User re-runs to get fresh snapshot.

### 5.3 Disk Space Limits

**Typical size:** 1 plan node × 3 expansion rounds = 3 × ~5 KB = 15 KB.

**Worst case:** 127 plan nodes × 5 rounds × 10 KB = 6.35 MB. Acceptable indefinitely.

**If grows unbounded (multi-repo scenario):**

```bash
# Future: cleanup command
roadmap cleanup --expansions --keep-last 3  # keep last 3 per parent node
```

For now: no cleanup. Size is not a blocker.

### 5.4 What if Expansion Script Fails to Execute?

**Current behavior (in `cmdExpand`):**

```typescript
try {
  execSync(`node --experimental-strip-types ${resolved}`, {...});
} catch (e) {
  throw new RoadmapError('VALIDATION_FAILED', {
    attempted: scriptPath,
    fix: 'Fix the expansion script and re-run',
  });
}
```

**User path:**
1. `roadmap complete <node>` → script written to stdout
2. User reads script (human review)
3. User runs `roadmap expand <path>`
4. Script fails (e.g., syntax error, bad JSON)
5. User gets clear error + suggestion to fix script
6. User edits `.roadmap/expansions/<nodeId>-<ts>.ts`
7. Re-runs `roadmap expand <path>` (same path)

Scripts are version-controlled after commit, so history is preserved even if they're edited.

## 7. Open Questions for Review

1. **Pre-expand dry-run?** Should `roadmap expand --dry-run` be part of this, or future?
   - Suggested: Future. Core design doesn't block it.

2. **Cleanup policy?** Keep all (current design) or auto-archive after N?
   - Suggested: Keep all now. Add cleanup later if needed.

3. **Script naming collision?** Could two concurrent agents write same filename?
   - Suggested: Timestamp is Unix seconds. If both within same second, second one gets `<nodeId>-<timestamp>-<sequence>.ts`. But rare enough to defer.

4. **Partial commits?** If expansion is only 50% of what's needed, can user commit partial DAG?
   - Suggested: No. Scripts are idempotent structural specs. Either the expansion is complete (all fix nodes for all failures), or it's not. This is checked by validate().

5. **Scriptable expansions (not intent-driven)?** Can users hand-write `.roadmap/expansions/my-custom.ts`?
   - Suggested: Yes! Same file format, same validate/commit workflow. writeExpansionScript() is a convenience; power users can author directly.

## 8. Examples

### Example 1: Intent-Driven Expansion (Happy Path)

```bash
$ roadmap complete plan-auth-strategy

# Validation fails: "JWT module implements rotation" at 0.42/0.90
# Script written: .roadmap/expansions/plan-auth-strategy-1709078400.ts

{
  "completed": false,
  "node": "plan-auth-strategy",
  "validation": {
    "passed": false,
    "expandable": true,
    "script": ".roadmap/expansions/plan-auth-strategy-1709078400.ts",
    "failedIntents": [{
      "statement": "JWT module implements rotation",
      "achieved": 0.42,
      "threshold": 0.90
    }],
    "nextStep": "Review then run: roadmap expand .roadmap/expansions/plan-auth-strategy-1709078400.ts"
  }
}

# User reviews script
$ cat .roadmap/expansions/plan-auth-strategy-1709078400.ts

# User runs expand
$ roadmap expand .roadmap/expansions/plan-auth-strategy-1709078400.ts --note "expanding auth module: JWT rotation implementation"

# Git commit: "roadmap: expand — +3 nodes via plan-auth-strategy-1709078400.ts"
# DAG now has: plan-auth-strategy-fix-0, plan-auth-strategy-fix-1, plan-auth-strategy-fix-2
```

### Example 2: Revision (Stalled, Depth Exceeded)

```bash
# First expansion, still not converged
$ roadmap expand .roadmap/expansions/plan-auth-strategy-1709078400.ts --note "first expansion: jwt rotation"

$ roadmap complete plan-auth-strategy-fix-0
# Still only 0.55/0.90. Expansion depth=2, stallThreshold=0.05 → converge check fails.

# Escalation script written: .roadmap/expansions/plan-auth-strategy-escalation-1709078402.ts
{
  "escalated": true,
  "reason": "depth-exceeded",
  "diagnosis": "Maximum expansion depth (2) reached without meeting threshold 0.90. Systematic issue likely requires different approach."
}

# User either:
# A) Reviews escalation script + runs roadmap expand
# B) Manually adjusts threshold or approach
# C) Escalates to human review
```

### Example 3: Concurrent Agents

```bash
# Agent A on approval-phase-2
$ roadmap complete approval-phase-2 --evaluate '[...]'
# Script A: .roadmap/expansions/approval-phase-2-1709078400.ts

# Agent B on runtime-tests (concurrent)
$ roadmap complete runtime-tests --evaluate '[...]'
# Script B: .roadmap/expansions/runtime-tests-1709078400.ts

# Both commit their expansions
$ roadmap expand .roadmap/expansions/approval-phase-2-1709078400.ts
$ roadmap expand .roadmap/expansions/runtime-tests-1709078400.ts

# DAG merges successfully (no collision)
```

## Summary

Expansion scripts are:

1. **Discoverable** — `.roadmap/expansions/<nodeId>-<timestamp>.ts` per invocation
2. **Readable** — human-friendly TypeScript with headers and section comments
3. **Traceable** — `_intentDiagnosis` + `expandedFrom` + git history
4. **Safe** — validated before commit, idempotent, replayable
5. **Concurrent-friendly** — timestamps prevent collisions
6. **Auditable** — full forensic trail from intent failure → diagnosis → script → committed DAG

Implementation is straightforward: new `expansion-writer.ts` module + integration point in `cmdComplete()`. No breaking changes to existing APIs.
